# 酒店预订管理系统 需求文档

> 版本：v1.0（初稿）　日期：2026-07-02
> 技术栈：单体 Spring Boot + MySQL + Vue3 + Element Plus

---

## 1. 项目概述

本系统是一个第三方酒店预订平台（类似携程），用户可以在平台上检索全国各地的酒店并预订客房。
系统覆盖：用户注册与积分体系、酒店/客房检索、客房预订与退订、订单管理、入住评价、汇总统计、后台信息维护。

**边界说明**：客人实际入住酒店后的具体管理（如前台办理、房卡、退房结算）不在本系统范围内，本系统只负责"预订"环节，订单状态由后台维护人员手工流转（已预订 → 已入住 → 已完成）。

## 2. 技术架构

### 2.1 总体架构

```
浏览器 (Vue3 + Element Plus)
        │ HTTP
        ▼
单体 Spring Boot 后端 app(8080)
  ├─ user    用户/积分/入住人
  ├─ hotel   酒店/房型维护与查询
  └─ order   订单/评价/统计
        │
      MySQL
```

- **后端形态**：单体 Spring Boot，前后端分离；前端通过 `/api` 代理访问 8080 后端。
- **数据库**：MySQL 8，单库多表（课程项目规模下不做分库）。
- **ORM**：MyBatis-Plus（简单增删改查零 SQL；视图/存储过程/统计等复杂查询写在 XML 里，保证 SQL 可见、可讲解）。

### 2.2 后端分层

| 层次 | 目录 | 职责 |
|---|---|---|
| Controller | `controller` | 接收 HTTP 请求、读取参数、做身份入口校验、返回统一结果 |
| Service | `service` | 处理业务规则，如注册登录、下单校验、订单流转、积分升级 |
| DAO/Mapper | `mapper` | 通过 MyBatis-Plus 和 XML SQL 访问数据库 |
| Entity/DTO | `entity` / `dto` | 映射数据库表、视图结果和请求对象 |

模块按业务包组织：`user`、`hotel`、`order`，共享返回结构和异常处理放在 `common`。


后端统一分层：`controller / service / mapper / entity`，一个类只负责一件事，控制单文件长度。

### 2.4 复杂度约定

复杂度预算向前端倾斜：

- **前端**：不设复杂度上限。为完整复刻博客的视觉与动效（卡片全屏展开、元素连续变形、飞出/飞回），允许编写精细的动画脚本，但仍须遵守目录职责划分（动效逻辑独立成模块，不混入页面组件）。
- **后端 Java**：按简单档定位，保持单体 Spring Boot，不引入注册中心、远程调用、复杂安全框架、Redis、MQ、分布式事务等中等档/复杂特性；一个 Controller 方法对应一个明确业务动作，避免过度抽象。
- **SQL**：表结构、视图、触发器、存储过程以"课程答辩时能逐行讲清楚"为标准，不用生僻语法，每个脚本文件短小、带注释。

## 3. 功能需求

### 3.1 用户管理

- **注册**：用户名、密码、手机号、身份证号（18 位校验）。密码用 BCrypt 加密后入库，只有注册用户才能预订。
- **登录**：登录后才能进入预订等功能页；未登录只能浏览酒店。后端简单登录过滤器校验 token 并注入当前用户身份。
- **信息维护**：用户可修改自己的手机号、密码等基本信息。
- **积分与 VIP**（自定策略）：
  - 订单完成（入住结束）后按实付金额累计积分：**1 元 = 1 分**。
  - 等级规则：积分 ≥ 1000 升级为 **VIP（预订 95 折）**；积分 ≥ 5000 升级为 **SVIP（预订 9 折）**。
  - 折扣在下单时按当前等级自动计算，订单同时记录原价和实付价。
- **常用入住人**：用户可维护一份入住人列表（姓名、身份证号、性别、出生日期、职业、受教育程度、收入状况），下单时直接勾选，这些信息也是"入住人画像统计"的数据来源。

### 3.2 酒店与客房检索

- 按 **城市、区域、酒店名（模糊）、价格范围、入住/离店日期** 组合查询，返回酒店清单。
- 点进某家酒店，展示该酒店全部房型：房型名、价格、可住人数、剩余可订数量（此查询通过视图 `v_hotel_room` 实现）。
- 日期条件参与"是否有房"的判断：某房型在所选日期段内剩余数量 > 0 才可预订。

### 3.3 客房预订

- 用户可为自己或他人预订，一次可预订 **多间、多种房型**。
- 每间客房必须登记入住人（从常用入住人中勾选或现场新增），一间房可登记多名入住人，但不得超过该房型的可住人数（如标间最多 2 人）。
- 下单时系统自动：校验日期合法性（离店 > 入住、不早于今天）、按 VIP 等级计算折扣、生成订单号。
- 预订成功后，该房型的可订数量自动减少（由触发器实现）。

### 3.4 订单管理

- 订单状态：**已预订 / 已入住 / 已完成 / 已退订**。
- 用户可按状态分类查询自己的订单，查看订单明细（酒店、房型、每间房的入住人、金额）。
- **退订**：状态为"已预订"（尚未入住）的订单可以退订，退订后房型可订数量自动恢复（触发器实现）。
- 后台管理员可查询所有订单，并流转订单状态（办理入住 → 已入住，离店 → 已完成，完成时给用户加积分）。

### 3.5 入住评价

- 状态为"已完成"的订单，用户可评价一次：评分（1~5 星）+ 文字评价。
- 酒店详情页展示该酒店的评分与评价列表。

### 3.6 汇总统计（报表：明细 + 汇总同时展示）

1. **客房预订情况统计**：按 城市 / 区域 / 酒店 / 价格区间 统计一段时间内的预订量、间夜数、金额，用于制定营销策略。
2. **入住人画像统计**：按入住人的 年龄段、性别、职业、受教育程度、收入状况 统计客房预订情况，分析不同客群的房型偏好。
3. **客房入住率统计**：统计一段时间内各类客房的入住率（存储过程实现）。

以上统计页面均以"汇总表 + 可展开的明细表"形式呈现。

### 3.7 后台信息维护

- 管理员账号（用户表中 role 区分）登录后可维护：酒店信息、房型与客房数量、订单状态、用户信息（如冻结）。
- 房型维护支持表格多选后的批量删除，满足批量操作要求。

## 4. 非功能需求

- **交互友好**：所有页面基于 Element Plus；日期用日期选择器（从根源杜绝 2021-5-35 这类非法日期）、性别用单选、城市/区域/职业等用下拉、入住人用复选，覆盖多种输入形式。
- **容错处理**：前端表单校验 + 后端参数校验双保险；后端统一异常处理，任何非法输入返回友好提示，程序不中断。
- **登录界面**：系统入口为登录页，未登录访问受限接口由后端过滤器拦截。
- **安全处理**：用户密码使用 BCrypt 加密；SQL 查询使用 MyBatis 参数绑定和 XML 占位符，避免字符串拼接造成 SQL 注入。
- **视觉与交互风格**：全站遵循第 7 节的前端风格规范（MD3 令牌配色、平铺无阴影、有迹可循的连续动效、组件复用）。

## 5. 数据库设计

### 5.1 概念设计（实体与联系）

实体：**用户**、**入住人**、**酒店**、**房型**、**订单**、**订单房间明细**、**评价**。

联系：
- 用户 1—n 入住人（常用入住人属于某用户）
- 酒店 1—n 房型
- 用户 1—n 订单；订单 n—1 酒店
- 订单 1—n 订单房间明细（一单多间）；明细 n—1 房型
- 订单房间明细 n—n 入住人（一间房多名入住人，用中间表）
- 订单 1—1 评价

> 简化说明：客房不落到具体房间号，只管理到"房型 + 数量"（total_count / available_count），足以支撑预订、退订与统计，避免引入房间排房逻辑。

### 5.3 完整性设计对照表（对应作业要求逐条落实）

| 要求 | 落实方式 |
|---|---|
| 主键/外键 | 每表均有 PK；t_guest/t_room_type/t_order/t_order_item/t_item_guest/t_review 均设 FK |
| 缺省约束 | gender 默认'男'；check_in_date 默认当天、created_at 默认 CURRENT_TIMESTAMP（MySQL 对应 GETDATE()）；**所有货币列（price/total_price/pay_price）统一默认 0** |
| 非空约束 | type_name（客房类型名）、username、hotel_name、city 等 |
| CHECK 约束 | check_out_date > check_in_date；score 1~5；points ≥ 0；available_count 范围 |
| 规则（身份证 18 位） | MySQL 无 SQL Server 的 RULE 对象，用 `CHECK (CHAR_LENGTH(id_card) = 18)` 等价实现 |
| 索引 | 见 5.4 |

### 5.4 索引设计

## 7. 前端风格规范（参照个人博客 TrueWebSite 的视觉与交互语言）

本系统前端必须复刻博客 `~/Program/TrueWebSite` 的风格。该风格可归纳为：**Material Design 3 令牌化配色 + 平铺无阴影 + 元素有迹可循的连续动效 + 严格的组件复用**。

### 7.1 设计令牌（唯一颜色来源）

全局只允许通过 CSS 变量取色，禁止在任何组件里写死色值。令牌沿用博客的 MD3 配色，并映射到 Element Plus 的主题变量（`--el-color-primary` 等）：

| 令牌 | 亮色 | 暗色 | 用途 |
|---|---|---|---|
| `--md-sys-color-primary` | `#0061a4` | `#9ecaef` | 主色：按钮、链接、标题重点词 |
| `--md-sys-color-primary-container` | `#d1e4ff` | `#00497d` | 主色容器：选中态、高亮背景 |
| `--md-sys-color-secondary` | `#535f70` | `#bbc7db` | 次要强调：标签文字 |
| `--md-sys-color-surface` | `#fdfcff` | `#1a1c1e` | 卡片表面 |
| `--md-sys-color-on-surface` | `#1a1c1e` | `#e2e2e6` | 主文字 |
| `--md-sys-color-on-surface-variant` | `#43474e` | `#c3c7cf` | 次要文字（描述、日期、辅助信息） |
| `--md-sys-color-background` | `#fdfcff` | `#1a1c1e` | 页面背景 |
| `--md-sys-color-outline` | `#73777f` | `#8d9199` | 描边（使用时降到 14%~24% 不透明度） |

暗色模式跟随系统 `prefers-color-scheme`，切换时背景/文字颜色带 `0.25s ease` 过渡。

### 7.2 平铺视觉规则

- **无阴影**：卡片、弹层一律不用 box-shadow 区分层次，用 1px 半透明描边（outline 色 × 14%~24% 不透明度，`color-mix` 实现）。
- **大圆角**：卡片 26px；标签、按钮为 999px 胶囊形（如博客的 tag 胶囊，本系统的房型标签、订单状态标签同款）。
- **字体排版**：`system-ui` 字体栈 + antialiased；页面级大标题用 `clamp()` 流式字号、行高 ≈ 0.96、负字距（-0.02em）、字重 500，标题中的关键词用主色着色（如首页 "预订全国的<span class=primary>酒店</span>"）；正文行高 1.7+；一切次要信息用 on-surface-variant。
- **布局**：内容区居中限宽（max-w-7xl），列表用响应式栅格（1 / 2 / 3 列随宽度切换），区块间距用统一的 gap 而非各自 margin。

### 7.3 交互原则："有迹可循"，元素不凭空出现

这是本风格的核心：**屏幕上出现的每个元素，都必须能追溯到它在上一个视图中的来源**，通过连续动效从来源"长"出来，而不是突然弹出。

- **展开代替跳转/弹窗**：酒店列表卡片点击后原地展开为酒店详情（卡片放大铺满内容区，标题、城市、价格等复用同一元素连续变大变位），返回时收回原处——对应博客的 post-card 展开阅读模式。订单列表 → 订单详情同理。动效实现可参照并移植博客的 `post-expander.js`（FLIP + WAAPI），允许做到同等精细度。
- **必须用弹层时从触发处生长**：确需 Dialog/Drawer 的场景（如新增入住人），弹层动效从触发按钮方向展开，且弹层内的标题要与触发按钮文字一致，让用户知道它从哪来。
- **统一动效参数**：缓动一律 `cubic-bezier(0.22, 1, 0.36, 1)`，时长 300~500ms；尊重 `prefers-reduced-motion`，用户关闭动画时全部动效退化为直接切换。
- **不打断布局**：滚动条用 `scrollbar-gutter: stable` 防止出现/消失时页面横跳。

### 7.4 组件复用规则

- 同一种信息只允许有一个组件实现：`HotelCard`（列表卡片/详情头部复用同一组件的两种状态）、`RoomTypeCard`、`OrderCard`、`GuestPicker`、`StatTable`（三张统计报表复用，汇总行可展开明细）、`StatusTag`（订单状态胶囊）等。
- 页面 = 组件的组合，`views/` 下的页面文件只做编排，不写具体展示逻辑；可复用展示逻辑全部沉到 `components/`。
- Element Plus 组件通过全局主题变量覆盖（圆角、主色、去阴影）统一到上述风格，禁止逐组件写内联样式修补。
